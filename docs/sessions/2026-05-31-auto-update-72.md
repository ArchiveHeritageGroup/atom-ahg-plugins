# Auto-Update / cron for client git pull (#72)

**Date:** 2026-05-31
**Issue:** ArchiveHeritageGroup/atom-extensions-catalog#72
**Repos:** atom-framework (bin/auto-update) + atom-ahg-plugins (ahgSettingsPlugin section)
**Status:** Implementation complete, lint-clean; script dry-run verified; settings route dispatch verified on PSIS; authenticated render pending.

## Scope

`bin/auto-update` already existed (pull framework+plugins, clear cache, --dry-run).
This closes the spec gaps: lock file, file logging, dirty-tree safety, rollback,
state file, notify, cron installer; plus a real settings section.

## Decisions (Johan)

- **Backup model:** git-only SHA rollback (no DB/file backup). Auto-update changes code,
  not data — capture pre-update SHA, reset back on failed validation.
- **Dirty working tree:** ABORT + warn (skip that repo). Never destroys client local edits
  (e.g. WDB's uncommitted theme mods).
- **Settings UI:** config + status + CLI page (no web "Run now"). Forced by ProtectSystem=full —
  php-fpm cannot run git on the /usr code tree, so runs happen via cron/CLI only.

## Changes

### atom-framework/bin/auto-update (hardened)
- `flock` concurrency guard; tee logging to `log/auto-update.log`.
- Per-repo dirty check → abort+warn (protects local edits).
- Capture pre-update SHA; on failed post-update validation (`php symfony cc`) →
  `git reset --hard` back to the previous commit for any repo that moved, re-clear cache.
- Writes `log/auto-update.state` (KEY=VALUE: LAST_RUN, RESULT, before/after SHAs, MESSAGE).
- Optional email notify (mail) on change / warning / rollback.
- `--check` (alias --dry-run); `--cron` (obeys AUTO_UPDATE_ENABLED toggle);
  `--install-cron` / `--uninstall-cron` (root; writes /etc/cron.d/atom-auto-update).
- Sources optional `log/auto-update.conf` (AUTO_UPDATE_ENABLED/FREQUENCY/NOTIFY_EMAIL).
- Bug caught in review: function logged to stdout AND echoed its return → callers captured
  log lines into SHA vars. Fixed to set globals (REPO_BEFORE/REPO_AFTER).

### atom-ahg-plugins/ahgSettingsPlugin — new "Auto-Update" section
- `handlers/autoUpdateAction.class.php` (SettingsAutoUpdateAction): saves enable/frequency/
  notify to `log/auto-update.conf` (+ mirrors to settings table); GET shows last-run status
  (from state file), the exact cron line, and CLI commands.
- `templates/autoUpdate.blade.php`: sidebar + status card + config form + cron/CLI instructions.
- Route `/ahgSettings/autoUpdate`; `executeAutoUpdate` delegate; menu entry "Auto-Update".

## Host constraint (documented, not a defect)

ProtectSystem=full on php8.3-fpm makes the /usr code tree read-only for the web worker, so
the web UI cannot run git — auto-update runs via cron/CLI only. A web "Run now" would need a
root-side trigger watcher (deferred; offered as an option).
